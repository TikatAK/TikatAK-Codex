# 计划书：Claude Code 自定义后端 AI 提供商改造

> 版本：v1.0 | 基于 claude-code 2.1.88 源码分析

---

## 一、可行性总结

**结论：可行，且比预期容易。**

源码中已存在完整的多提供商架构（Bedrock / Vertex / Azure Foundry / firstParty），且支持 `ANTHROPIC_BASE_URL` 环境变量覆盖 API 端点。改造核心在于新增一个 `custom` 提供商类型，并配套 UI/CLI 设置界面。

---

## 二、现有架构分析

| 组件 | 文件 | 现状 |
|------|------|------|
| 提供商选择 | `utils/model/providers.ts` | 已支持 firstParty / bedrock / vertex / foundry，通过环境变量切换 |
| API 客户端 | `services/api/client.ts` | 每个提供商对应不同 SDK 实例化逻辑，已有 `baseURL` 注入点 |
| API Key 存储 | `utils/auth.ts` | 支持环境变量 → FD → apiKeyHelper 脚本 → macOS 钥匙串 → config 文件，优先级完整 |
| 模型配置 | `utils/model/configs.ts` | 模型 ID 按提供商映射，支持 `modelOverrides` 自定义覆盖 |
| 基础 URL | `constants/oauth.ts` | `ANTHROPIC_BASE_URL` 已可覆盖 firstParty 端点 |
| 自定义 Header | `services/api/client.ts` | `ANTHROPIC_CUSTOM_HEADERS` 已支持任意 HTTP 头注入 |

**关键发现：** 由于大多数第三方 AI 提供商（OpenAI、DeepSeek、Qwen、Mistral、月之暗面、智谱等）都提供 **OpenAI 兼容接口**，而 Anthropic SDK 本身支持自定义 `baseURL`，最简路径是：在 `firstParty` 模式下注入自定义端点 + API Key，无需替换 SDK。

但为了完整支持非 Anthropic 消息格式（不同的 stop_reason、content_block 结构等），还需要一层适配层。

---

## 三、改造方案（三档可选）

### 方案 A — 轻量级（最快落地，2~3 天）

**思路：** 利用现有 `ANTHROPIC_BASE_URL` + `ANTHROPIC_API_KEY` + `ANTHROPIC_MODEL` 三个环境变量，配合中间代理服务（如 LiteLLM、one-api、new-api）将任意提供商转为 Anthropic 兼容协议，**源码零修改**。

```
用户设置 → 代理服务 → 任意 AI 提供商
                ↑
        Anthropic 兼容协议
```

**工作量：**
- 集成一个自托管代理（如 LiteLLM Docker）
- 在 Claude Code 设置界面（或 .env 文件）添加三个字段的配置 UI

**限制：** 需要额外运行代理服务，对用户有一定门槛。

---

### 方案 B — 中量级原生支持（推荐，1~2 周）

**思路：** 在源码中新增 `custom` 提供商类型，支持：
- 自定义 `baseURL`
- 自定义 `apiKey`
- 自定义 `modelId`
- 可选自定义 `headers`

对于 OpenAI 兼容接口，复用现有 firstParty SDK 并修改端点；对于不兼容接口，增加格式适配器。

**核心改动文件：**

```
utils/model/providers.ts        ← 新增 'custom' 类型
services/api/client.ts          ← 新增 custom provider 实例化分支
utils/auth.ts                   ← 新增 customApiKey 存储/读取
utils/model/configs.ts          ← 支持完全自定义 modelId
constants/oauth.ts              ← 新增 CUSTOM_BASE_URL 常量
settings/settings.ts            ← 新增 customProvider 配置项
commands/config.ts              ← 新增 /provider 配置命令
```

**新增环境变量：**
```bash
CLAUDE_CODE_USE_CUSTOM_PROVIDER=true
CUSTOM_PROVIDER_BASE_URL=https://api.openai.com/v1
CUSTOM_PROVIDER_API_KEY=sk-xxx
CUSTOM_PROVIDER_MODEL=gpt-4o
CUSTOM_PROVIDER_API_FORMAT=openai   # openai | anthropic
```

**新增 settings.json 字段：**
```json
{
  "customProvider": {
    "enabled": true,
    "baseURL": "https://api.deepseek.com/v1",
    "apiKey": "sk-xxx",
    "model": "deepseek-chat",
    "apiFormat": "openai",
    "headers": {}
  }
}
```

**消息格式适配器（针对 OpenAI 格式）：**
- 请求：Anthropic Messages → OpenAI Chat Completions
- 响应：OpenAI SSE chunks → Anthropic stream events
- 工具调用：`tool_use` ↔ `function_call` / `tool_calls` 双向转换

---

### 方案 C — 重量级完整抽象（2~4 周）

**思路：** 将整个 API 层重构为 `ProviderAdapter` 接口，每个提供商实现独立适配器，未来可扩展。

```typescript
interface ProviderAdapter {
  sendMessage(params: MessageParams): AsyncIterable<MessageEvent>
  validateApiKey(key: string): Promise<boolean>
  listModels(): Promise<ModelInfo[]>
}

class AnthropicAdapter implements ProviderAdapter { ... }
class OpenAIAdapter implements ProviderAdapter { ... }
class CustomAdapter implements ProviderAdapter { ... }
```

**优点：** 最彻底，易于长期维护和扩展。  
**缺点：** 改动最大，风险最高，维护成本高。

---

## 四、推荐路径：方案 B（分阶段实施）

### 阶段一：基础配置支持（第 1~3 天）

- [ ] `providers.ts` 新增 `custom` 类型及检测逻辑
- [ ] `settings.ts` 新增 `customProvider` 配置 schema
- [ ] `auth.ts` 新增 custom API Key 的存/读/删逻辑
- [ ] `client.ts` 新增 custom provider 的 SDK 实例化（复用 firstParty SDK + 自定义 baseURL）
- [ ] 支持通过 `/config` 命令交互式设置提供商参数

### 阶段二：OpenAI 格式适配（第 4~7 天）

- [ ] 实现 `OpenAIMessageAdapter`：请求/响应格式双向转换
- [ ] 处理流式输出（SSE chunk 格式差异）
- [ ] 工具调用格式适配（function_call ↔ tool_use）
- [ ] 错误响应格式适配（rate limit / auth error 统一化）

### 阶段三：用户体验（第 8~10 天）

- [ ] CLI 命令：`claude provider set` / `claude provider status` / `claude provider test`
- [ ] 首次启动引导：检测到无 Anthropic Key 时提示配置自定义提供商
- [ ] 连接测试：配置保存时验证 API 可达性
- [ ] 错误提示优化：区分认证错误 / 网络错误 / 模型不存在

## 五、主要技术挑战

| 挑战 | 难度 | 说明 |
|------|------|------|
| OpenAI 工具调用格式转换 | ★★★★ | parallel tool_use vs tool_calls 结构差异复杂 |
| 流式响应格式差异 | ★★★ | content_block_delta vs delta.content 结构不同 |
| OAuth 登录流程 | ★★★ | 自定义提供商无 Anthropic OAuth，需改为纯 API Key 模式 |
| 模型能力差异 | ★★ | 部分功能（如 extended thinking）仅 Claude 支持，需降级处理 |
| 价格/用量统计 | ★★ | token 计数单位可能不同，影响显示 |

---

## 六、不影响的功能（无需改动）

- Bash、FileEdit、Grep、MCP 等所有工具实现
- 多 Agent 协调（coordinator）
- Vim 模式、Voice 模式
- 会话管理、历史记录
- 项目配置（.claude/config.json）
- 除 API 通信外的所有 commands

---

## 七、关键风险

1. **协议差异风险**：部分 OpenAI 兼容 API 实现不完整，工具调用可能有细节差异，需逐一测试。
2. **维护成本**：原始包升级时，需同步 merge 改动（建议 fork 维护）。
3. **功能降级**：Extended thinking、文件上传等 Anthropic 专有功能在第三方提供商上不可用，需做能力检测和优雅降级。
4. **许可证问题**：源码为非官方还原版，商业部署前需评估法律风险。

---

## 八、结论

**强烈推荐采用方案 B**。

- 改动相对有限（约 7~10 个文件），不触及工具/命令核心逻辑
- 可复用现有的多提供商基础设施
- 通过环境变量和 settings.json 配置，兼容现有使用习惯
- 支持 OpenAI 兼容接口意味着可以接入绝大多数第三方提供商（DeepSeek、月之暗面 Kimi、智谱 GLM、Qwen 等）

预计总工时：**8~12 个工作日**（独立开发者），可保留 Claude Code 所有核心功能。
